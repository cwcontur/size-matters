How it works

A breakdown of the extension's architecture and key implementation details.

Size Matters is built entirely with Google Apps Script — no external servers, no build tools, no dependencies beyond what Google provides. The extension consists of a handful of .gs server-side files and a set of HTML template files that form the sidebar UI. Here's how each piece fits together.

Server-side (.gs)
appscript.json Main.gs Margins.gs Reset.gs Colours.gs CurrentPageSettings.gs Cached.gs

Runs on Google's servers. Interacts with the Docs API. Manages user-scoped persistence via PropertiesService.

→
Client-side (HTML)
Sidebar.html SidebarCSS.html SidebarUtils.html SidebarLogic.html SidebarColorPicker.html

Runs in the sidebar iframe. Manages UI state, input validation, and calls the server via google.script.run.

appscript.json — project manifest

The manifest declares the V8 runtime and the two OAuth scopes the extension uses. Nothing more is requested — documents.currentonly grants access only to the currently open document, and script.container.ui allows the sidebar panel to be shown. 

"oauthScopes": [
  "https://www.googleapis.com/auth/documents.currentonly",
  "https://www.googleapis.com/auth/script.container.ui"
]

Main.gs — entry points

onInstall() and onOpen() are the standard Apps Script lifecycle hooks. They add a "Size Matters" menu item to the Google Docs toolbar. showSidebar() compiles the HTML template and opens it as a sidebar panel.

The include() helper is used by the HTML templates to stitch together multiple files at render time — so SidebarCSS.html, SidebarUtils.html, and so on are all inlined into the final sidebar output via <?!= include('filename') ?> template tags.

function include(filename) {
  return HtmlService.createHtmlOutputFromFile(filename).getContent();
}

Cached.gs — user persistence

All user preferences are stored with PropertiesService.getUserProperties(), which is scoped per-user and persists across sessions. Three things are saved: the preferred unit of measurement, custom page size presets (capped at 20), and custom saved colors (capped at 30).

Custom presets are stored as a JSON-serialized object keyed by a sanitized version of the preset name. Before saving, the backend enforces the 20-preset cap and throws a catchable MAX_LIMIT_REACHED error if it's hit — so the frontend can show an appropriate warning without crashing. 

// Preset keys are generated like this:
const safeKey = "custom_" + name.toLowerCase().replace(/[^a-z0-9]/g, "");

// The limit check:
if (!presets[safeKey] && Object.keys(presets).length >= MAX_PRESETS) {
  throw new Error("MAX_LIMIT_REACHED");
}

Colours.gs — page background color

The Google Docs API represents colors as RGB floats between 0 and 1, not the 0–255 integers or hex strings that a browser color picker works with. This file handles the conversion boundary. setPageColor() accepts an RGB object with float components and writes it to body.setBackgroundColor(), which expects a hex string — so it converts on the way in. getCurrentPageColor() reads the hex string back from the API and converts it to floats for the frontend. 

// Server-side: floats (0–1) in, hex string to the API
function rgbToHex(r, g, b) {
  const toHex = (c) => {
    const hex = Math.round(c * 255).toString(16);
    return hex.length === 1 ? "0" + hex : hex;
  };
  return "#" + toHex(r) + toHex(g) + toHex(b);
}

Note: the client side has its own rgbToHex/hexToRgb functions that work with 0–255 integers instead — the two sides of the bridge use different scales by design.

CurrentPageSettings.gs — reading document state

The Docs API always returns page dimensions in points (1/72 of an inch). getCurrentPageSettings(unit) fetches width, height, and all four margins, then converts them to whatever unit the user has selected before returning the data.

getInitialSidebarData() is the most important function for startup performance — it bundles the user's unit preference, their saved custom presets, and the current page settings into a single server call, so the sidebar can fully initialize in one round trip instead of three.

function getInitialSidebarData() {
  const unit = getUserUnitPreference() || "inches";
  return {
    customPresets: getCustomPresets(),
    unitPreference: unit,
    pageSettings: getCurrentPageSettings(unit),
  };
}

Margins.gs — writing page settings

setCustomPageSizeWithMargins(settings, unit) is the core write operation. It receives all six values (width, height, four margins) already converted to points by the frontend, then calls the six corresponding body.set*() methods on the document. Errors are caught, logged, and re-thrown — so the client-side withFailureHandler fires and shows an error toast rather than silently failing.

Reset.gs — restoring defaults

Google Docs' default page is US Letter: 8.5 × 11 inches with 1-inch margins on all sides. In points (the unit the API uses), that's 612 × 792 with margins of 72. The reset function hardcodes these values directly.

body.setPageHeight(11 * 72);  // 792pt
body.setPageWidth(8.5 * 72);  // 612pt
body.setMarginTop(72);        // 1 inch on all sides
body.setMarginBottom(72);
body.setMarginLeft(72);
body.setMarginRight(72);

SidebarUtils.html — shared client utilities

This file loads first (included before the logic files) and defines the shared building blocks everything else depends on. A few worth highlighting:

CONVERSIONS — a lookup object with toPoints() and fromPoints() for inches, cm, and points. Using a single central object means unit conversion logic is never duplicated.

const CONVERSIONS = {
  inches: { toPoints: (v) => v * 72,         fromPoints: (v) => v / 72 },
  cm:     { toPoints: (v) => (v / 2.54) * 72, fromPoints: (v) => (v * 2.54) / 72 },
  points: { toPoints: (v) => v,               fromPoints: (v) => v }
};

compareFloats() — floating-point arithmetic means 8.5 * 72 doesn't always equal exactly 612. This helper wraps Math.abs(a - b) <= tolerance and is used everywhere a numeric comparison needs to be reliable.

ToastManager — a single object that handles all toast notifications. All toast messages are defined in a TOAST_TYPES config object (including support for message functions that take a unit string), so adding a new notification never requires touching the display logic.

The file also includes test suites for unit conversion and color math that run automatically when CONFIG.debug is set to true.

SidebarLogic.html — core application logic

The largest file. All page state is stored in AppState, with all internal values kept in points regardless of what unit the user has selected. This means conversions only happen at the display layer — when reading from inputs or writing to them — never during computation.

Input validation happens in a debounced handler that fires 300ms after the user stops typing. It checks for out-of-range values (enforcing a 0.1–120 inch safe zone) and also catches margin overlap — where combined left + right or top + bottom margins would leave no room for content.

Orientation swap is a good example of the logic being careful. Switching from portrait to landscape doesn't just swap width and height — it also transposes top/bottom margins with left/right margins, which is the correct behavior for a rotated page.

// Portrait → Landscape: swap dimensions AND transpose margins
inputs.width.value  = vals.h.toFixed(2);  // old height becomes new width
inputs.height.value = vals.w.toFixed(2);
inputs.top.value    = vals.l.toFixed(2);  // old left becomes new top
inputs.left.value   = vals.t.toFixed(2);
inputs.bottom.value = vals.r.toFixed(2);
inputs.right.value  = vals.b.toFixed(2);

Custom preset saving has several layers of protection: duplicate dimension detection (before the modal even opens), overwrite confirmation (if the same preset name is reused), and a 20-preset hard cap enforced on both the client and server independently.

Unit switching converts the live input values — not the saved internal state — so if a user has typed a modified value that they haven't applied yet, that value is preserved and converted correctly when they switch units.

SidebarColorPicker.html — the color picker

The gradient canvas is drawn pixel-by-pixel using ImageData rather than CSS gradients, which gives exact HSL control. The X axis is saturation (0→1), the Y axis is lightness (1→0), and the hue comes from the slider above.

for (let y = 0; y < height; y++) {
  for (let x = 0; x < width; x++) {
    const saturation = x / (width - 1);
    const lightness  = 1 - (y / (height - 1));
    const [r, g, b]  = hslToRgb(hue, saturation, lightness);
    // write r,g,b,255 into ImageData at (x,y)...
  }
}

The hex input has a ghost-text prediction system: as you type, a translucent overlay input shows a predicted completed hex value. Pressing Tab accepts the prediction. Single-character input predicts by repeating the character (f → ffffff); three-character input expands shorthand (abc → aabbcc).

Custom colors use optimistic UI updates — the color swatch appears in the grid immediately on click, and the server save happens asynchronously in the background. If the save fails, an error toast appears but the UI has already responded, making the interaction feel instant.

SidebarCSS.html — styling

All colors are defined as CSS custom properties at the :root level, organized by role (text, backgrounds, borders, buttons, toasts, etc.). This makes the entire color scheme easy to adjust from one place and keeps component styles from hardcoding values.

The signature entrance animation — where sidebar elements drop in with a random rotation and a slight bounce — is a single @keyframes hard-slam rule applied via the .tumble class, with the rotation angle randomized per-element via a CSS variable set by JavaScript on load.

@keyframes hard-slam {
  0%   { opacity: 0; transform: translateY(-800px) rotate(var(--random-rotation, 10deg)); }
  75%  { opacity: 1; transform: translateY(0) rotate(0deg); }
  85%  { transform: translateY(3px); }   /* subtle impact overshoot */
  100% { transform: translateY(0); }
}

Frequently asked questions

Why doesn't Google Docs support custom page sizes natively?

No one knows. Google Docs has offered a fixed set of presets for years with no indication this will change. Size Matters exists to fill that gap by applying page dimensions directly to your document using the Google Docs API.

Is this add-on really free?

Yes. Completely free, no subscription, no trial period. Setting a custom page size is a basic feature — it shouldn't cost money.

How do I open the Size Matters sidebar?

In Google Docs, go to Extensions → Size Matters → Open. The sidebar will appear on the right side of your document.

What units of measurement are supported?

You can enter dimensions in inches, centimeters, or points. Switch between them using the units selector at the top of the sidebar — the values will convert automatically.

What page size presets are included?

The preset dropdown includes common sizes like Letter (US), A4, A5, Legal, Tabloid, and more. You can also type in any custom dimensions you need.

Will changing the page size affect my text and content?

Yes — changing page dimensions will reflow your document content to fit the new size, the same as it would in any word processor. It's a good idea to check your layout after applying a new size.

How do I reset to the default Google Docs page size?

Click the Reset to default button in the sidebar. This restores the standard Letter size (8.5 × 11 in), default margins, and white page color.

Are my saved colors stored anywhere?

Custom colors you save are cached locally within Google Apps Script's PropertiesService, associated with your Google account. They are never sent to any external server.

The add-on isn't appearing in my Extensions menu. What do I do?

Try refreshing Google Docs. If it still doesn't appear, go to Extensions → Add-ons → Manage add-ons and make sure Size Matters is installed and enabled. If you continue to have issues, email us.
